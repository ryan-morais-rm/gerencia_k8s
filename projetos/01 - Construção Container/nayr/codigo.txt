=======================================================================
DOCUMENTAÇÃO DO CÓDIGO FONTE - MOTOR DE CONTAINER (NAYR)
=======================================================================

1. VISÃO GERAL DAS BIBLIOTECAS (IMPORTS)
-----------------------------------------------------------------------
- `std::fs` e `std::path::Path`: Bibliotecas nativas do Rust para manipular caminhos, arquivos e deletar/criar pastas no disco.
- `std::process::Command`: Executa comandos externos (subprocessos). Usado para rodar o "git", executar o próprio binário de novo 
   (re-exec) e rodar os comandos finais dentro do container.
- `clap::{Parser, Subcommand}`: Cria a interface de linha de comando (CLI) profissional, traduzindo o que o usuário digita em estruturas Rust.
- `nix::mount::{mount, MsFlags}`: Faz a ponte direta com o Kernel do Linux para a syscall `mount`.
- `nix::sched::{unshare, CloneFlags}`: A mágica dos containers. Isola recursos do sistema operacional criando Namespaces (PID, Mount, Hostname, etc).
- `rusqlite::{Connection, Result}`: Biblioteca de banco de dados para conectar, ler e gravar no SQLite nativo.

2. ESTRUTURA DA CLI (Struct Cli e Enum Commands)
-----------------------------------------------------------------------
Define os comandos que o usuário pode digitar:
- `Pull`: Baixa a imagem base de um repositório git.
- `Run`: Inicia o container com um nome e um comando (padrão: `/bin/sh`).
- `Ps`: Lista os containers registrados no banco de dados.
- `Rm`: Remove os arquivos temporários e o registro do banco de dados de um container.
- `Child` (Oculto): O usuário nunca digita isso. O programa usa esse comando internamente para continuar rodando *depois* de 
   ter entrado na "Matrix" (Namespaces isolados).

3. O FLUXO PRINCIPAL: `main()`
-----------------------------------------------------------------------
A validação inicial verifica se o usuário é `root` (UID 0), pois o Kernel Linux exige permissões máximas para usar namespaces e 
montagens especiais. Após isso, o programa lê qual comando foi digitado:

A. COMANDO PULL (`nayr pull`)
   Chama a função auxiliar para clonar ou atualizar (via git) o sistema de arquivos base na pasta `images/base`.

B. COMANDO RUN (`nayr run`) - "O Processo Pai"
   1. Cria Pastas: Chama `preparar_pastas_overlay` apenas para criar as pastas vazias no host. Ele NÃO faz a montagem aqui para não "sujar" a máquina real.
   2. Isolamento (Unshare): Cria uma barreira no Kernel separando processos (NEWPID), montagens (NEWNS) e hostname (NEWUTS).
   3. O Re-exec: O binário chama a si mesmo (`/proc/self/exe`) passando o comando oculto `child`. Esse novo processo "filho" nasce 
      inteiramente dentro do ambiente isolado.
   4. Banco de Dados (Início): Pega o PID (ID do processo) do filho e insere no banco SQLite com o status "Running".
   5. Espera: O Pai fica bloqueado esperando o Filho (o container) terminar. Quando acaba, atualiza o SQLite para "Exited".

C. COMANDO CHILD (`nayr child`) - "A Matrix"
   Roda dentro do ambiente isolado e prepara o terreno final:
   1. Montagem Privada: Aplica a flag `MS_PRIVATE` na raiz. Isso impede que as montagens que faremos a seguir vazem para o Vivobook real.
   2. Mount do OverlayFS: Agora sim, chama `montar_overlay_interno` para juntar as pastas em camadas de forma segura. Quando o container 
      morrer, esse mount morre com ele.
   3. Hostname e Proc: Altera o hostname e monta o sistema `/proc` falso, para que comandos como `ps` executados *dentro* do container só vejam
      os processos isolados.
   4. A Prisão (Chroot): Tranca a visão do sistema para a pasta do container.
   5. Execução Segura: Pega a string do comando digitada pelo usuário (ex: "ls -la") e quebra em pedaços lógicos com `split_whitespace`. 
      Separa o programa ("ls") dos argumentos ("-la"). Executa o programa e entrega o controle ao usuário.

D. COMANDO PS (`nayr ps`)
   Conecta no banco `container.db`, executa um `SELECT` puxando o nome, PID, status e comando. Itera sobre os resultados e imprime uma tabela 
   visualmente formatada na tela.

E. COMANDO RM (`nayr rm`)
   1. Apaga os Arquivos: Encontra as pastas `overlays/nome` e `containers/nome` criadas durante a execução e usa `fs::remove_dir_all` para varrê-las do disco.
   2. Limpa o DB: Faz um comando `DELETE` no SQLite buscando o nome do container para remover o seu registro.

4. FUNÇÕES AUXILIARES DETALHADAS
-----------------------------------------------------------------------
- `baixar_ou_atualizar_imagem`: Orquestra processos filhos do sistema executando `git clone` ou `git pull` para baixar o rootfs (Alpine, Debian, etc).
- `preparar_pastas_overlay`: Apenas cria a infraestrutura de pastas físicas (`upper`, `work`, e a pasta de mesclagem) exigidas pelo OverlayFS sem montar nada.
- `montar_overlay_interno`: Monta uma string de opções de montagem (`lowerdir=X,upperdir=Y,workdir=Z`) e faz a syscall `mount` com a flag 
   "overlay", sobrepondo os arquivos temporários em cima da base imutável.
- `init_db`: Cria o arquivo físico do banco de dados na raiz do projeto e emite um `CREATE TABLE IF NOT EXISTS`, garantindo que a tabela de containers 
  (nome, pid, status, comando) esteja sempre pronta para ser usada sem crashs.
