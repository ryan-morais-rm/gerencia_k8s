=======================================================================
DOCUMENTAÇÃO DO CÓDIGO FONTE - MOTOR DE CONTAINER (NAYR)
=======================================================================

1. VISÃO GERAL DAS BIBLIOTECAS (IMPORTS)
-----------------------------------------------------------------------
- `clap::{Parser, Subcommand}`: É a biblioteca responsável por criar a interface de linha de comando (CLI). 
   Ela lê o que o usuário digita no terminal (ex: "nayr run --name teste") e transforma isso em variáveis seguras no Rust.
- `nix::mount::{mount, MsFlags}`: Faz a ponte entre o Rust e o Kernel do Linux para executar a chamada de sistema (syscall) `mount`. 
   Usado para montar o OverlayFS e configurar diretórios privados.
- `nix::sched::{unshare, CloneFlags}`: A peça central dos containers. A função `unshare` serve para "desvincular" o processo atual da máquina real, 
   criando os Namespaces (isolamento).
- `std::fs` e `std::path::Path`: Bibliotecas padrão do Rust para manipular caminhos de pastas e criar diretórios no disco.
- `std::process::Command`: Usado para executar comandos externos (subprocessos) a partir do Rust. Usamos para rodar o "git", 
   para o binário chamar a si mesmo (re-exec) e para rodar o "/bin/sh" final.

2. ESTRUTURA DA LINHA DE COMANDO (STRUCT CLI E ENUM COMMANDS)
-----------------------------------------------------------------------
O programa define três comandos principais usando o `clap`:
- `Pull`: Recebe uma URL de repositório (`--repo`) para baixar a imagem base.
- `Run`: Recebe o nome do container (`--name`) e o processo que vai rodar (`--exec`, com valor padrão de `/bin/sh`).
- `Child`: É um comando OCULTO (`hide = true`). O usuário não deve digitar isso. Ele é usado internamente pelo próprio 
  programa para continuar a execução depois que o ambiente é isolado.

3. A FUNÇÃO PRINCIPAL: `main()`
-----------------------------------------------------------------------
A execução começa aqui e segue estas etapas lógicas:

A. Validação de Root: 
   O programa verifica se está rodando como `root` (UID 0). Sem isso, o Kernel proíbe criar Namespaces e fazer montagens.

B. O Comando 'Pull':
   Se o usuário digitou `pull`, chama a função `baixar_ou_atualizar_imagem` definindo o destino como "images/base".

C. O Comando 'Run' (O Processo Pai):
   Se o usuário digitou `run`, o "Processo Pai" faz 3 coisas:
   1. Chama o OverlayFS: Cria as pastas upper/work e junta com a imagem base.
   2. Isola o Ambiente: Executa `unshare` isolando PID (processos), UTS (rede/hostname) e Mount (pontos de montagem).
   3. O Truque do Re-exec: O Processo Pai cria um "Processo Filho" chamando seu próprio binário (`/proc/self/exe`) 
      mas passando o comando oculto `child`. Em seguida, o Pai fica "esperando" (`wait`) esse Filho terminar.

D. O Comando 'Child' (O Processo Filho Isolado):
   Quando o programa roda como `child`, ele JÁ ESTÁ DENTRO do ambiente isolado (Namespaces). Aqui ocorre a "mágica" final do container:
   1. Proteção de Montagem: Roda um `mount` com a flag `MS_PRIVATE` na raiz (`/`). Isso impede que as montagens 
      que o container fizer afetem a máquina real.
   2. Hostname: Altera o nome da máquina (dentro do container) para o nome passado pelo usuário.
   3. Montagem do /proc: Monta o sistema de processos virtual (`/proc`) para que comandos como `ps` 
      funcionem dentro do container enxergando apenas os processos isolados.
   4. A Prisão (Chroot): Executa `chroot` mudando o diretório raiz (`/`) para a pasta `containers/nome_do_container`.
   5. O Executável Final: Roda o comando pedido pelo usuário (ex: `/bin/sh`). Neste momento, o terminal do usuário é anexado ao container.

4. FUNÇÕES AUXILIARES
-----------------------------------------------------------------------
A. `baixar_ou_atualizar_imagem(repo_url, caminho_destino)`
   - Verifica se a pasta "images/base" já existe.
   - Se já tiver um ".git" lá dentro, significa que a imagem já foi baixada. O Rust então dispara no terminal um comando 
     `git pull` para atualizá-la.
   - Se não existir, o Rust dispara um `git clone <URL>` para baixar o sistema de arquivos base inteiro.

B. `configurar_e_montar_overlay(name)`
   - É responsável por preparar o "disco rígido" do container.
   - Verifica se a imagem base realmente existe antes de continuar.
   - Cria 3 pastas dinâmicas específicas para o container atual:
     * `overlays/{nome}/upper`: Onde as alterações (escrita) serão salvas.
     * `overlays/{nome}/work`: Pasta de controle exigida pelo Kernel do Linux.
     * `containers/{nome}`: O diretório final (Merged).
   - Monta a string de opções exigida pelo Kernel ("lowerdir=...,upperdir=...,workdir=...").
   - Dispara a chamada de sistema nativa `mount` do tipo "overlay", unindo as pastas de leitura e escrita e projetando o 
     resultado na pasta `containers/{nome}`.
   - Retorna o caminho da pasta montada para que a função principal saiba onde aplicar o `chroot`.

=======================================================================
RESUMO DO FLUXO (O CICLO DE VIDA)
=======================================================================
Usuário digita: nayr run -> nayr monta o OverlayFS -> nayr cria os Namespaces -> nayr clona a si mesmo (child) -> nayr child configura 
diretórios privados -> nayr child aplica o chroot -> nayr child executa o /bin/sh -> Usuário ganha o shell isolado.
