=======================================================================
DOCUMENTAÇÃO DO CÓDIGO FONTE - MOTOR DE CONTAINER (NAYR)
=======================================================================

1. VISÃO GERAL DAS BIBLIOTECAS (IMPORTS)
-----------------------------------------------------------------------
- `std::fs`, `std::path::Path` e `std::fs::OpenOptions`: Bibliotecas nativas do Rust para manipulação do sistema de arquivos do Host. 
  Permitem criar diretórios (cgroups, overlayfs), validar caminhos (volumes), apagar recursivamente (rm) e gerenciar descritores de arquivos para redirecionamento de logs de saída (stdout/stderr).
- `std::process::{Command, Stdio}`: API para spawnar e gerenciar subprocessos. Usada para invocar comandos externos (Git), 
  disparar o Re-exec do próprio binário (`/proc/self/exe`), lançar o processo final no jail e redirecionar fluxos de entrada/saída (Stdio) dinamicamente.
- `clap::{Parser, Subcommand}`: Framework robusto para parsing de argumentos (CLI). Transforma comandos e flags em structs Rust.
- `nix::mount::{mount, umount, MsFlags}`: Bindings POSIX para syscalls nativas de montagem. Usadas para OverlayFS, Bind Mounts (Volumes), procfs e desmontagem segura.
- `nix::sched::{unshare, CloneFlags}`: O coração da virtualização a nível de SO. A syscall `unshare` cria os novos Namespaces (PID, Mount, UTS) do Kernel.
- `nix::unistd::Pid` e `nix::sys::signal::{kill, Signal}`: Utilizados para gerenciamento de processos POSIX, permitindo o envio de sinais de interrupção (ex: SIGTERM) nativos ao SO.
- `rusqlite::{Connection, Result}`: Driver do banco SQLite local. Persiste metadados do ciclo de vida, incluindo estado, PID atual e comando base do container.

2. ESTRUTURA DA CLI (Struct Cli e Enum Commands)
-----------------------------------------------------------------------
Mapeia de forma explícita as instruções expostas na interface de console:
- `Pull`: Sincroniza o RootFS imutável via Git (`--repo`).
- `Run`: Cria e inicializa um container a partir do zero. Aceita restrições (`--memory`), mapeamento de diretórios do host (`--volume`), comando alvo (`--exec`) e uma flag para anexar o terminal do usuário (`--interactive`).
- `Start`: Retoma a execução de um container previamente parado, recuperando seu último comando do banco de dados e recriando seus namespaces. Suporta religação de volumes (`--volume`).
- `Stop`: Interrompe graciosamente um container em execução enviando um sinal SIGTERM ao seu processo principal.
- `Ps`: Lista o estado histórico e em tempo real dos containers catalogados.
- `Logs`: Recupera e exibe o histórico de saída padrão (stdout/stderr) gravado durante a execução em background do container.
- `Rm`: Remove todas as persistências, pontos de montagem e metadados associados à instância.
- `Child` (Oculto): Comando interno acionado programaticamente pelo pai para assumir a execução pós-ruptura de namespaces.

3. O FLUXO PRINCIPAL: `main()`
-----------------------------------------------------------------------
A execução inicia validando privilégios de superusuário (`root` / UID 0) via `nix`. Sem isso, o Kernel bloqueia a criação de Namespaces e pontos de montagem.

A. COMANDO PULL (`nayr pull`)
   Aciona `baixar_ou_atualizar_imagem` para clonar o repositório base para `images/base`.

B. COMANDO RUN (`nayr run`) - "Criação e Execução Inicial"
   Este bloco atua no nível do Host:
   1. Preparação de Disco: Cria a estrutura OverlayFS (`preparar_pastas_overlay`) e a infraestrutura do ficheiro de logs em `logs/<nome>/container.log`.
   2. Ruptura de Namespaces (`unshare`): Cria barreiras para PID (processos ocultos), Mount (arquivos isolados) e UTS (hostname independente).
   3. Roteamento de Stdio: Verifica a flag `--interactive`. Se ativa, herda (`Stdio::inherit`) o terminal do usuário (stdin/out/err). Se inativa (background), redireciona a saída para os descritores de arquivo do log criado e anula a entrada (`Stdio::null`).
   4. Mecanismo de Re-exec: O processo pai executa `/proc/self/exe child` (fork), passando as flags de `--name`, `--exec` e injetando a flag `--volume` se fornecida.
   5. Cgroups e Banco de Dados: O pai captura o PID do filho, aplica restrições (`configurar_cgroup`), insere o registro inicial no SQLite (Status: "Running") e aguarda (`.wait()`) o fim da execução para atualizar para "Exited".

C. COMANDO START (`nayr start`) - "Retomada de Execução"
   1. Validação: Consulta o SQLite. Se o container estiver "Running", aborta. Caso contrário, resgata o comando (`stored_cmd`) que foi passado no `Run` original.
   2. Recriação: O processo de ruptura de namespaces, abertura do ficheiro de log em modo *append* e roteamento de I/O é repetido.
   3. Re-exec: O filho é lançado novamente usando o comando resgatado e a flag de volume, recebendo um novo PID dinâmico do sistema. O banco é atualizado.

D. COMANDO STOP (`nayr stop`) - "Interrupção"
   1. Localização: Busca o PID atual do container no banco de dados, desde que o status seja "Running".
   2. Sinal POSIX: Utiliza `nix::sys::signal::kill` para enviar um `SIGTERM` diretamente ao processo mestre do container, solicitando encerramento suave.
   3. Atualização: Marca o status do container como "Exited" na base de dados.

E. COMANDO CHILD (`nayr child`) - "Dentro do Jail"
   Roda isolado nos Namespaces:
   1. Blindagem (`MS_PRIVATE`): Isola a árvore de montagem para não refletir no Host.
   2. Fusão OverlayFS: Une a imagem base e a camada volátil.
   3. Bind Mounts (Volumes): Se a flag `--volume` foi recebida, invoca `montar_volume` para criar um elo direto entre a pasta real do Host e um diretório correspondente dentro da camada isolada `merged_path`.
   4. Pseudo-FS e Identidade: Monta `/proc` local e altera o hostname.
   5. Chroot: Redireciona a raiz lógica (`/`) para a pasta fundida e aplica `set_current_dir("/")`.
   6. Execução: Processa a string de comando (split_whitespace) e invoca o binário alvo como processo líder do container.

F. COMANDO LOGS (`nayr logs`)
   Acessa o ficheiro de texto em `logs/<nome>/container.log` (gravado no modo background) e despeja seu conteúdo no terminal.

G. COMANDOS PS & RM (`nayr ps` / `nayr rm`)
   - `Ps`: Lista os dados dinâmicos lidos do SQLite.
   - `Rm`: Limpeza completa, que agora inclui uma verificação de segurança importante: invoca `desmontar_volume` (umount) na pasta do container ANTES de aplicar o `fs::remove_dir_all`. Isso evita o erro de "Device or resource busy" e impede a exclusão acidental de dados reais do Host que estivessem atrelados via volume. Além do OverlayFS e BD, também remove a pasta do log e o limite de Cgroup.

4. DETALHAMENTO DE FUNÇÕES AUXILIARES E CGROUPS
-----------------------------------------------------------------------
- `configurar_cgroup`: Manipula o **Cgroups v2** gravando limites em `memory.max` e injetando o PID em `cgroup.procs`.
- `montar_volume(vol_str, merged_path)`: Faz o parse da string `Host:Container` do usuário. Valida a existência da origem no Host, garante a existência do destino dentro do jail (`fs::create_dir_all`) e executa um `mount` com a flag `MS_BIND`, projetando a pasta de fora para dentro.
- `desmontar_volume`: Garante a liberação segura (umount) de pontos de montagem pendentes no ciclo de limpeza (Rm) do Host.
- `baixar_ou_atualizar_imagem`: Clone ou pull incremental do RootFS base.
- `preparar_pastas_overlay` & `montar_overlay_interno`: Arquitetura em 3 camadas do UnionFS (`lowerdir`, `upperdir`, `workdir`).
- `init_db`: Inicialização idempotente do banco relacional DDL (SQLite).
